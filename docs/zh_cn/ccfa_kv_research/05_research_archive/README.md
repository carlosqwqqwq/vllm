# 调研过程与结果沉淀

这部分的目标不是重复前面的结论，而是把这轮围绕 `vLLM + KV cache + 系统类 CCF-A 选题` 的**调研方法、证据链和判断过程**保存下来，方便后续读者直接复用。

## 1. 为什么单独保留这一部分

很多研究笔记只留下最终结论，例如“建议做 `TierKV`”或“`AffinityKV` 很有潜力”。但如果没有保留过程，后面的人会缺少三样最关键的东西：

1. **事实基础**：这些判断到底建立在哪些源码、文档、PR 和 issue 之上。
2. **筛选标准**：为什么有些方向被保留，有些方向被降级。
3. **可复用方法**：下次别人继续调研时，应该先看哪、再验证哪、最后怎么收敛。

所以这里保留的是“**调研工作流**”，不是普通结论摘要。

## 2. 这一轮调研回答了什么问题

本轮调研围绕三个主问题展开：

1. 当前 `vLLM` 在 KV cache 上究竟发展到了什么阶段。
2. GitHub 社区最近一年真正推进了什么，以及哪些问题仍未解决。
3. 如果目标是系统类 CCF-A，哪些选题最值得做，且能兼容当前 `vLLM`。

## 3. 建议阅读顺序

如果你是第一次接手这个方向，建议这样读：

1. 先看 `调研过程与方法`
   - 理解这轮是怎么建立事实基础的
2. 再看 `调研结果与判断`
   - 先建立对全局的判断
3. 然后回到上一级的：
   - `01_vllm_status`
   - `02_github_progress`
   - `04_refined_ideas`

## 4. 这部分适合谁看

- 刚接手 `vLLM` / KV 方向的同学
- 需要复盘本轮选题逻辑的人
- 后续准备继续扩展论文主线的人
- 想把调研过程变成实验计划的人

## 5. 后续如何继续补

建议以后继续沿着同样结构迭代：

1. 新增事实时，优先补到“过程”文档里的证据来源章节
2. 判断变化时，优先补到“结果”文档里的排序与推荐章节
3. 原始材料增多时，优先补到“原始证据索引”
4. 需要解释“为什么这个证据支持这个方向”时，补到“证据到选题的映射”
5. 新 idea 成熟后，再补到 `04_refined_ideas/`

这样能保证文档长期可维护，而不是一次性笔记。
