# 1 引言

前缀复用已经成为现代大模型推理系统的基础能力。无论是长 system prompt 的多轮对话、共享工具描述的工具增强请求，还是模板稳定的批量推理，只要运行时能够复用已经计算完成的前缀状态，就可以显著降低首 token 延迟并提升有效吞吐。因此，`vLLM` 等主流推理框架都把 prefix caching 视为核心运行时机制，而不是附属优化。

然而，现代在线负载早已不再满足“共享前缀 + 单一接口 + 单一路径执行”的理想假设。两个请求即使拥有相同的输入前缀，也可能具有不同的返回语义、输入形态或执行路径。于是，系统能否复用已有状态，不再只取决于“这段前缀是否已经被计算过”，还取决于当前请求能否在其消费条件下合法使用这份状态。现有 serving runtime 因而主要优化已经发生的命中，却缺少机制去回答一个更根本的问题：参考执行条件下本应成立的复用机会，为什么没有在当前请求中兑现为物理命中收益？

我们在 `vLLM` 上的请求级测量表明，这并不是零散的兼容性问题，而是会系统性吞掉运行时收益的共性现象。在 `MT-Bench W1` 中，普通生成请求可以稳定获得前缀命中；但当请求额外要求返回输入 token 的概率信息时，这部分命中会直接下降为 `0`。在公开的 `LongBench W1` 负载上，这一现象进一步放大：在筛选后的 `64` 个请求、`32` 个 family 上，仅增加额外观测需求，就会使 `483504` 个原本可复用的前缀 token 全部失去实际命中。类似地，同一组前缀在一种执行后端中能够稳定复用，在另一种执行后端中却明显失去复用收益。表面上看，这些现象来自不同路径；实质上，它们揭示的是同一个运行时缺口，即系统没有显式建模“哪些既有状态在当前请求下仍然可消费”。

本文将这种“逻辑上应命中、物理上却未命中”的现象形式化为 `false-negative exact hit`。它不是一般意义上的缓存未命中，也不是简单的工程兼容性缺陷，而是一类更具系统意义的运行时失效：请求在参考执行条件下已经具备复用条件，却没有在当前执行中得到相应的物理命中收益。这里所谓“逻辑上应命中”，并不意味着当前缓存表中必然已有一个可直接读取的物理条目，而是意味着语义等价的前缀计算已经在兼容参考执行条件下被证明可复用；相应地，这里的失败也不意味着 hash 无法定位前缀，而是意味着系统缺少一层显式机制，去区分“状态存在”“当前可消费”和“当前值得恢复”这三件不同的事。

围绕这一观察，本文依次开展请求级测量、契约建模与系统验证。我们首先在 `vLLM` 中构建请求级测量框架，以区分参考执行条件下可复用的前缀计算与当前请求实际兑现的复用收益；在此基础上，本文进一步刻画既有状态与当前请求之间的契约关系，并据此设计 `TwinPathServe`，使运行时仅在安全条件满足且恢复代价可接受时恢复复用，否则保守回退。原型结果表明，这一思路不仅能够解释两类代表性的失效来源，也能够在相应场景中恢复原本丢失的前缀收益并降低延迟。

本文的贡献可以概括为三点。

1. 我们提出并形式化了大模型推理中的 `false-negative exact hit` 问题，把 `logical hit`、`physical hit`、`false-negative hit` 与 `bridgeable miss` 组织为一组可测量、可复核的运行时概念，并指出前缀复用的失败并不总是由状态不存在或 hash 未命中造成，也可能源于参考执行条件下已证明可复用的 state 未能在当前 consumer contract 下兑现为物理命中收益。
2. 我们提出一种契约感知的分析与设计框架，用以解释为什么既有前缀状态在不同请求条件下未必仍然可被直接消费；`TwinPathServe` 是这一框架的第一版系统实例。
3. 我们给出两个代表性场景下的原型证据，表明 observation-related mismatch 与 backend-related mismatch 都能够在保守的正确性边界内恢复部分原本丢失的复用收益。
