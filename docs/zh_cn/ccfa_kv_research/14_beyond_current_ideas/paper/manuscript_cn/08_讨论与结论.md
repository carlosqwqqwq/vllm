# 8 讨论与结论

## 8.1 统一抽象的必要性

`prompt_logprobs selective replay` 与 twin-pool route 在实现层面差异明显，但二者仍需要被置于统一抽象之下。第七章已经表明，现有近邻工作分别沿着 physical-hit optimization、match relaxation、state-medium redesign 和 workflow continuity 等路线推进，而本文的问题落在这些路线之间的空白处。若将本文拆解为两个局部 patch，问题将被重新解释为 feature compatibility fix 或局部 runtime heuristic，从而失去其作为独立系统问题的结构性内涵。

之所以需要统一抽象，并不是因为统一表述本身更简洁，而是因为 `W1` 到 `W5` 的共同结构无法由分散 patch 充分解释。`prompt_logprobs` 与 backend mismatch 的实现路径虽不相同，但它们都表现为同一类现象：reference execution 下已经成立的 exact 或 logical-exact reuse，没有在当前 request 到来时兑现为 `physical hit`。如果缺少 `reuse contract` 这样的中介层，系统只能在 API 特判、backend 黑名单、保守 skip 与局部 fallback 之间零散修补，而无法解释为何这些表面上彼此无关的 miss 会在 mixed-API serving 中反复出现。因此，本文的贡献不在于把两条路径分别修通，而在于把分散的 capability boundary 上升为可测量、可归因、可恢复的统一 runtime pathology。

## 8.2 一般性、可证伪性与安全边界

本文当前的主证据仍然来自 `vLLM`，因此结论始终锚定在 `vLLM` 这一实例系统上。但这并不意味着问题只存在于 `vLLM`。只要一个 serving runtime 同时具备三类条件，它就有可能暴露相似的 `false-negative exact hit`：其一，系统内部存在 shared-state reuse；其二，用户侧暴露异构 API 契约与 observation requirement；其三，执行面存在异构 backend path 或 capability class。本文对一般性主张保持克制，并非因为这一结构不存在于其他系统，而是因为当前最强的 request-level measurement、prototype 结果与 recovery evidence 仍集中在 `vLLM`。

`SGLang` 的 supporting evidence 可以进一步说明这种“克制的一般性”。在 `SGLang` 中，已经可以观察到多类与本文同方向的 capability boundary：`input_embeds` 要求关闭 radix cache，多模态 reranker 文档建议关闭 radix cache，部分 deterministic 或 backend-specific path 也会弱化 cache compatibility。这些现象说明，prefix identity 并不是复用的充分条件，consumer path 与 backend path 的可消费性边界是现代 serving runtime 的一般结构之一。但这些 supporting evidence 仍不足以替代跨框架主实验。就目前证据而言，更合适的表述是：`vLLM` 是被严格测量和原型化的实例系统，而 capability-boundary 这一结构在其他 runtime 中也存在可观察迹象。

本文的另一条核心边界，是安全主张始终针对 `soundness` 而非 `completeness`。本文所证明的不是“已有 state 都应被复用”，也不是“只要 hash 命中就应直接消费”；本文所证明的是：只有当 `SafeReuseContract` 与局部 bridge 前提同时成立时，系统才允许恢复，否则必须 fallback。第六章的两类 prototype 结果进一步支持了这一边界。对 `prompt_logprobs selective replay` 而言，sidecar、boundary replay 与 replay budget 都是 bridge 成功的必要条件，而且 `32-family` formal validation 已经表明该 bridge 能把默认零命中恢复到接近完整的 `effective reuse`；对 twin-pool route 而言，恢复方式不是跨不兼容 backend 直接共享 state，而是把 family 重新送回兼容执行面，且 `MT-Bench W5` formal evaluation 已经表明它能够在公开 workload 上把 degraded hit 恢复到 reference-compatible 水平。因此，本文并不是取消现有 runtime 的保护，而是在区分：哪些保护是 correctness 的底线，哪些隔离仅仅反映系统暂时缺少 bridge。

更严格地说，本文当前已经完成的是“条件化 soundness”论证。理论层面，第四章给出的命题依赖 `SafeReuseContract` 的各个谓词能够被在线正确验证；实验层面，第六章已经给出接口级 smoke、必要条件消融和 fallback 边界，但尚未形成覆盖 top-k token、rank、logprob 数值逐项对齐以及长尾 fallback path 的完整 output-equivalence audit matrix。因此，当前更恰当的结论是：理论给出安全恢复的充分条件，prototype 提供初步符合这些条件的运行证据，而完整的输出等价性审计仍有待后续工作补齐。

最后，本文保持明确的可证伪性。若后续大规模公开 benchmark 证明 `false-negative exact hit` 只在极少数工作负载中出现，或统一 substrate 无法比 scattered fixes 提供更强的解释力与系统收益，那么本文的贡献就应被重新界定。同样，若第二类 bridge 只能在极窄的合成场景中成立，而无法在 representative workload 上恢复 degraded hit，`TwinPathServe` 的系统意义也会明显减弱。换言之，本文提出的是一条可以被后续实验支持或推翻的运行时命题。

## 8.3 已证实的结论与当前局限

基于第三章和第六章，本文已经能够给出三个相对稳固的结论。第一，`false-negative exact hit` 并不是对 prefix miss 的重新命名，而是一类具有 request-level evidence 的 runtime pathology。第二，这类 pathology 不是单点 feature bug，因为 `W1` 与 `W5` 已经分别展示了 observation mismatch 与 backend compatibility mismatch 如何在不同路径上暴露同一种 `logical hit -> physical hit` 断裂。第三，两类 bridge 都已超出设计稿阶段：`prompt_logprobs selective replay` 已在 `32-family` formal validation 中恢复 `effective reuse`，而 `TwinPathServe` 双池原型也已在 `MT-Bench W5` 的公开 workload 上把 backend mismatch family 重新送回兼容执行面，并恢复 degraded hit。

本文同时仍有明确局限。首先，`W5` 在 vanilla runtime 一侧仍主要表现为 cross-backend logical false-negative evidence，而不是在线原生 `backend_mismatch` 归因已经闭环；当前更强的结果来自 twin-pool runtime 的公开 workload 恢复证据。其次，虽然已经出现 prototype-level 恢复结果，但多 benchmark、统一 baseline matrix、`p95/p99 TTFT`、steady-state throughput、route fallback rate 与 overload behavior 仍未在 representative workload 上系统报告。再次，对第一类 bridge 而言，当前 formal 结果仍来自 dedicated validation runner，而不是已经接进统一 public runner 的 full-method 路径；覆盖 top-k token、rank、logprob 数值一致性以及长尾 fallback case 的 output-equivalence 审计也还没有完整展开。最后，尽管第七章已经表明本文与 PIC、hybrid cache、workflow continuity 等近邻路线边界清楚，但“统一 substrate 相对 best ad hoc fix 的全面经验优势”仍需要更完整的公开实验支持。也就是说，当前文稿已经足以回答“为什么这是一个独立系统问题”和“为什么这类 bridge 可以在安全边界内成立”，但尚不足以支持“完整系统已经在所有目标 workload 上稳定优于所有替代方案”这一更强结论。

## 8.4 结论

本文提出 `FalseNegativeKV`，把现代大模型推理框架中一类长期存在却缺乏清晰命名的现象明确化为系统问题：某些请求在 reference execution condition 下已经构成 exact 或 logical-exact reuse，却在计算开始前没有被安全兑现为自己的 `physical hit`。围绕这一现象，本文首先给出了 measurement-first 的观测面与 `logical hit / physical hit / false-negative hit / bridgeable miss` 的统一问题表征；随后提出 `reuse contract`、`SafeReuseContract`、`false_negative_reason` 与 `bridge operator` 这套契约感知抽象；最后以 `TwinPathServe` 给出第一版 `soundness-first` runtime design。

与现有 prefix-cache、shared-prefix execution、PIC、hybrid cache 或 workflow continuity 相关工作相比，本文填补的空白并不是“如何更快地利用已经成立的命中”，也不是“如何让更多不再精确对齐的上下文继续复用”，而是：当一份既有计算已经在参考执行条件下被证明可复用时，现代 serving runtime 为什么仍然不能把它安全稳定地兑现为当前 request 的命中收益。当前 prototype 结果已经表明，这一问题并不只能停留在 measurement 或解释层面；在 `prompt_logprobs` 和 backend mismatch 这两类强实例上，contract-aware bridge 已经能够在保守边界内恢复一部分原本会被 runtime 直接丢掉的命中。

因此，本文的结论并不是“所有 prefix miss 都能被修复”，也不是“已有 state 就应当被任意复用”。更恰当的结论是：在现代大模型推理系统中，exact-prefix reuse 的关键问题已经不再只是 identity lookup，而是 reference execution 下的既有计算，能否在当前 consumer contract 中被安全兑现为实际收益。一旦这层 runtime gap 被显式建模，系统才有可能从 scattered compatibility fixes 走向统一、轻量且可验证的命中恢复能力。
