# 4 契约感知复用抽象

第三章给出了两类强测量证据。其一，在 `W1` 中，额外的 observation 需求会把原本稳定的 exact-prefix reuse 直接打成零命中；其二，在 `W5` 中，同一 `prompt family` 在一个 backend compatibility class 中可以稳定复用，但切换到另一类执行条件后，逻辑命中机会会在运行前完全丢失。表面上看，这两个现象分别对应 API 语义差异与 backend 路由差异；从运行时角度看，它们揭示的是同一结构性缺失，即系统缺少一套显式语言来刻画“哪些 producer state 在哪些 consumer 条件下仍然可消费”。

本章据此提出契约感知的复用抽象。该抽象需要与第三章的测量框架保持一致，能够在同一语义空间中容纳 `logical hit`、`physical hit` 与 `false_negative_reason`，统一解释 `W1` 与 `W5`，并为后续系统设计提供区分“应保守回退”与“可在明确条件下恢复”的操作性边界。

## 4.1 从测量现象到统一问题表征

如果只从 feature 视角出发，`prompt_logprobs` 兼容性问题、backend compatibility 切换问题、`prompt_embeds` 与 multimodal 的 carrier 问题看起来彼此并无共同点。前者表现为 observation API 的局部限制，后者表现为 backend 或输入载体引入的新边界。然而，第三章的测量已经说明，系统真正失去的不是某项功能支持，而是参考执行条件下已经建立的 exact reuse 没有在当前请求中兑现为物理命中。

因此，本文不再把问题组织成“某条路径不支持 APC”或“某个 backend 没命中”之类的局部缺陷列表，而是将 exact reuse 的系统语义分解为三层。第一层是问题表征层，用以回答哪些请求在逻辑上应命中、哪些请求在物理上真正命中，以及两者之间损失了多少收益。第二层是归因层，用以解释这些收益为何没有被兑现。第三层是恢复层，用以回答哪些失效在当前系统边界内可以被安全 bridge，哪些则必须继续视为 out-of-scope。

## 4.2 四个核心概念

本文的第一层问题表征由四个核心概念构成：`logical hit`、`physical hit`、`false-negative hit` 和 `bridgeable miss`。

`logical hit` 指请求在逻辑上本可复用的前缀状态。它强调的是“兼容参考执行已经证明这段前缀可复用”，而不是当前单次 lookup 的执行结果；更准确地说，它是围绕 compatible reference execution 构造的 counterfactual reuse oracle，而不是脱离执行条件的绝对 ground truth。`physical hit` 指运行时在当前请求中真实兑现的前缀复用量。`false-negative hit` 则表示前两者之间的差值，即逻辑上已具备复用条件，但在运行前未被兑现为物理命中的那一部分。最后，`bridgeable miss` 指 `false-negative hit` 中理论上可以在当前研究边界内通过特定 bridge operator 恢复的那一部分。

这四个概念分别回答不同问题。`logical hit` 回答“在 compatible reference execution 下本应可兑现多少”；`physical hit` 回答“运行时实际只命中了多少”；`false-negative hit` 回答“损失了多少原本可兑现的收益”；`bridgeable miss` 则把问题进一步收缩为“哪些收益值得进入当前系统恢复面”。其中：

```text
false-negative hit = max(logical hit - physical hit, 0)
bridgeable miss ⊆ false-negative hit
```

第三章的 `W1` 与 `W5` 都不是“普通 miss”，而是满足 `logical hit > 0`、但 `physical hit = 0` 的强实例。如果缺少这一层问题表征，后续章节就无法解释为什么 `W1` 与 `W5` 属于同一问题类，也无法解释为何某些 miss 值得恢复、某些则不值得。

## 4.3 `reuse contract`

为了让上述分解在运行时真正可用，本文把 `reuse contract` 定义为 producer state 与 consumer requirements 之间的一组显式关系。它至少需要回答三个问题：第一，producer 侧到底发布了什么可复用状态；第二，consumer 侧需要哪些额外能力才能消费这些状态；第三，当两者不匹配时，系统应把 miss 归为哪一类 mismatch，以及是否存在对应的 bridge operator。

这个定义的关键，在于它把“状态存在”与“收益兑现”之间原本隐含的那一步显式化了。hash、prefix key 或 radix structure 只能帮助系统回答“是不是同一个前缀身份”，却不能单独回答“当前请求是否仍能合法消费这份状态”。`reuse contract` 正是用来补上这一步的语言层。

在当前主线中，`reuse contract` 最直接地统一解释了第三章的两类强证据。对 `W1` 而言，producer 发布的是 prefix KV，但 consumer 额外需要 prompt-level observation；因此问题不在状态不存在，而在 observation contract 没有被满足。对 `W5` 而言，参考执行类中已经证明这段前缀可复用，但当前请求落入了不同的 backend compatibility class；因此问题也不在“两个池之间没有共享缓存”这么浅层的常识，而在 consumer 所在执行类并不具备消费这份 state 的能力。

这一抽象还有一个重要后果：它把“是否恢复”从 ad hoc patch 提升为统一决策。只要一个 miss 能被表述为 producer-state / consumer-requirement mismatch，系统就可以进一步判断这一 mismatch 是否已经被 request-level trace 显式捕获，以及它是否存在安全可验证的 bridge。就此而言，`reuse contract` 并非对若干工程 case 的重新命名，而是从测量走向系统设计所必需的中介层。

不过，`reuse contract` 只回答“某个 miss 是否具备被恢复的资格”，尚未回答“当前是否值得恢复、应该以何种代价恢复”。对许多 counterfactual hit 而言，bridgeability 与 runtime value 并不等价：有些 miss 虽然理论上可 bridge，但 observation sidecar 不完整、boundary replay 成本过高，或者把请求重新送回兼容执行面会带来不可接受的排队代价。因此，在 `reuse contract` 之后还需要引入一层 `realization policy`，用于比较 bridge gain 与 bridge cost，并把候选 miss 映射为 `direct_reuse`、`observation_bridge`、`backend_route` 或 `measure_only` 等 `realization_mode`。第四章的核心含义由此可以概括为：先用 contract 判定可恢复资格，再用 realization policy 判定是否值得在当前运行条件下兑现。

## 4.4 `SafeReuseContract` 与 `soundness`

安全性是契约感知复用成立的基本前提。`FalseNegativeKV` 并不主张“已有 state 一旦存在就应被直接复用”，也不主张“只要命中 hash 就是安全的”。未经契约验证的复用可能影响输出，这正是现有系统在若干路径上保守禁用 prefix reuse 的原因。本文的安全主张因此不是 `completeness`，而是 `soundness`：本文给出的是**安全复用的充分条件**，而不是所有复用机会的完全识别。

为此，本文把安全复用进一步收束为一组更严格的条件，可记为 `SafeReuseContract`。对一个请求而言，它至少需要同时满足下面五点。第一，`PrefixIdentity`：被复用的 prefix state 必须对应于与当前请求一致的 exact 或 logical-exact 前缀，而不是 family 对齐失败或 cache identity 漂移。第二，`ExecutionCompatibility`：producer state 必须产生于当前 consumer 支持的 execution / backend compatibility class；如果 backend class 不兼容，系统不得直接消费这份状态。第三，`CarrierCompatibility`：当前 consumer 所需的输入载体语义必须与 producer 发布的 state 相容，例如 token-id-sensitive consumer 不能在仅有 embeds carrier 时假定自己可以正确复用。第四，`ObservationCompleteness`：当前 consumer 所需的 observation 必须被完整覆盖；若裸 KV 不足以支持当前请求的返回语义，则系统必须引入 sidecar、replay 或其他等价 bridge。第五，`VerifiedFallback`：只要上述任一条件不成立、无法在线验证、或代价超预算，系统就必须回退到 baseline 支持路径。

这五个条件共同定义了本文的 `soundness-first` 设计原则。本文宁可保守地错过一个本可安全 bridge 的 case，也不接受一个不安全的请求被错误复用。由此带来两个直接后果：其一，后续系统设计必须把 fallback 视为契约的一部分，而不是实现细节；其二，本文所有“恢复”主张都应理解为“在 `SafeReuseContract` 成立时的安全恢复”，而不是“对所有 logical hit 的完备恢复”。

进一步地，本文将上述五个条件收束为一个条件性命题。**命题 1（contract-aware bridge 的 soundness）**：若请求 `r` 在进入 bridge 前满足 `SafeReuseContract(r)`，且对应 bridge 的局部前提也成立，则系统执行 bridge 后的用户可见输出与对应 baseline 支持路径等价；若任一条件不成立、无法在线验证或代价超预算，系统必须 fallback 到 baseline，因此不会产生 `unsafe false positive reuse`。

这一命题的证明思路如下：`PrefixIdentity` 排除错误前缀别名，`ExecutionCompatibility` 与 `CarrierCompatibility` 排除 consumer 非法消费，`ObservationCompleteness` 保证不会因为省算而缺失用户可见字段，`VerifiedFallback` 则把所有不确定情形收束回 baseline。于是，系统只会在“身份正确、消费合法、观测完整、失败可退回”的前提下执行 bridge。对 `prompt_logprobs selective replay` 而言，baseline 是 `full-prefill + full prompt observation`；对 twin-pool routing 而言，baseline 是进入兼容 execution class 或直接 full fallback，而不是跨不兼容 backend 直接共享 state。由此可见，本文的理论能证明的是 bridge 决策的 `soundness`，而不是复用机会识别的 `completeness`。

命题 1 给出的是真正意义上的条件性 `soundness` 论证。其含义在于：若 `SafeReuseContract` 的谓词能够被正确、完备且在线地验证，则 bridge 决策在逻辑上是 sound 的；它并不等同于声称当前实现已经完成了所有谓词的在线完备判定，也不等同于声称本文已经对所有接口、所有 top-k / rank / logprob 字段完成了输出等价性审计。第六章给出的实证结果主要覆盖接口级 observation completeness、fallback correctness 与显著时延收益；更完整的 output-equivalence audit 仍属于后续扩展范围。

## 4.5 `false_negative_reason` 与 `bridge operator`

在这个抽象里，`false_negative_reason` 不是为了给每个 miss 随意贴标签，而是为了把运行时归因和后续恢复放入同一语义空间。一个合理的 `false_negative_reason` taxonomy 至少应具备两点：第一，它必须与第三章的 measurement surface 对齐，能够覆盖主导性 miss 来源，而不是把大部分案例都丢进 `unknown`；第二，它必须与 `bridge operator` 发生对应关系，也就是说，归因为何种 mismatch 的请求，应当能映射到一类明确的恢复路径，或者明确归为当前 out-of-scope。

基于这一要求，本文当前把 `prompt_logprobs_skip` 视为 observation mismatch 的强实例，把 cross-backend `backend_incompatibility` 视为 backend mismatch 的强实例。二者都已经在第三章获得了 request-level 或 family-level 的测量支撑，因此进入 P1 恢复面。相比之下，`carrier_incompatibility`、`partial_prefill_incompatibility` 与更广泛的 pooling 相关 mismatch 虽然同样符合这一抽象语言，但在当前正文中仍主要承担 supporting 或边界论证角色，而不是第一版主恢复目标。

因此，本文在统一抽象下首先选择两类证据最充分的 mismatch 作为第一批 bridge operator。这里的“统一”并不意味着所有 mismatch 必须共享同一个实现，而是意味着它们需要共享同一套 measurement、reasoning 与 recovery 语言。只有在这一前提下，后续的 `TwinPathServe` 才能构成一个 runtime substrate，而不是若干彼此独立的 feature patch。

## 4.6 统一抽象相对于离散修补的优势

对 `prompt_logprobs` 与 backend mismatch 分别修补，确实可以缓解局部问题，但这不足以解释为何两类失效都表现为 exact reuse 在计算开始前丢失，也不足以判断未来出现的新 mismatch 是否属于同一问题族。统一抽象的价值，正在于将这些表面差异较大的现象重新组织为同一结构性缺口。

统一 substrate 至少带来三层收益。第一，它提供解释力：第三章中看似来自不同路径的 `W1` 与 `W5`，现在都可以被表述为 producer state 与 consumer requirement 之间的 contract mismatch。第二，它提供测量闭环：`logical hit`、`physical hit`、`false_negative_reason` 与 `bridgeable miss` 不再是彼此脱节的指标，而是共享同一套语义框架。第三，它直接支撑系统控制面：一旦 mismatch 被放入统一 contract 语言中，后续系统就可以围绕 route key、bridge operator 与 fallback 形成一致的 admission 与 recovery 逻辑。

这种“统一”并不要求两类 bridge 共享同一个实现函数，而是要求它们共享同一条运行时控制链。在本文当前 prototype 中，两类 bridge 已经共享同一组 request-level trace 字段、同一套 `reuse contract` 判断、同一类 fallback 可见性，以及同一条 `reuse oracle -> contract gate -> bridge / route -> trace` 的控制顺序。统一 substrate 的价值因此不在于把所有修复写成单个 patch，而在于把原本分散的策略判断提升为同一个 correctness-aware reuse control plane。

统一抽象的意义在于将共享原因显式化：现代大模型推理系统中的很多命中丢失，并非由状态不存在或单次 hash 未命中直接造成，而是源于系统缺少一套统一语言来表达 producer state、consumer requirement、mismatch 原因与恢复机制之间的关系。下一章将在这一抽象基础上给出其在 `TwinPathServe` 中的第一版系统化实例。
