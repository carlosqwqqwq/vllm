# 7 相关工作

与本文最接近的研究分布在通用 serving runtime、分布式调度与路由、shared-prefix execution、非 exact-prefix 复用、状态介质扩展以及 workflow continuity 等方向。这些工作共同表明，前缀相关状态管理已经成为大模型推理系统的重要议题。然而，各条路线对复用问题的基本假设并不相同：有的默认命中已经成立，关注命中后的执行效率；有的主动放宽匹配条件；有的更换状态介质；还有的延长状态生命周期。本文关注的则是另一类问题：在 reference execution condition 下已经成立的 exact 或 logical-exact reuse，为何未能在当前 consumer contract 下转化为 `physical hit`。

## 7.1 通用 LLM Serving Runtime：优化已经成立的命中

现代大模型 serving runtime 已经形成较为稳定的基础结构。`Orca` 将 iteration-level scheduling 与 selective batching 引入生成式 Transformer serving，奠定了现代 autoregressive serving 的调度起点。`vLLM` 通过 `PagedAttention`、continuous batching 与面向 serving 的内存管理，显著提升了长上下文和高并发场景下的推理效率；`SGLang`、`DeepSpeed-FastGen`、`TGI` 和 `TensorRT-LLM` 等系统，也分别从 radix cache、runtime DSL、工业级高吞吐推理栈与编译/内核优化等角度推动了 serving runtime 的成熟。

这些系统为本文提供了直接的实例语境：shared-state reuse、typed API、异构 backend 和复杂 execution path 已经成为现代 serving runtime 的常态。然而，它们在问题假设上与本文仍有根本差异。现有通用 serving runtime 的核心任务，大多是提高已经成立的 `physical hit` 的利用效率，或在命中成立后更快地执行 prefill / decode。换言之，这些工作主要回答的是“如何更高效地利用已经可消费的 state”，而不是“为什么 logical hit 没有在运行前 materialize 为当前 request 的命中收益”。

因此，本文与这些工作既保持连续，也保持边界。没有 `Orca`、`vLLM` 和 `SGLang` 这类 runtime，本文关注的问题不会以如此集中的方式出现；但本文的研究对象并不是另一套“更快的 serving runtime”。`FalseNegativeKV` 关注的是这些 runtime 在 mixed-API、mixed-backend 场景中尚未显式建模的一层：`producer state` 与 `consumer requirement` 之间的可消费性契约。

## 7.2 调度、解耦与分布式 prefix-aware routing：默认 hit 语义已经成立

另一批接近的工作来自调度、解耦和分布式 serving 路线。`Sarathi-Serve` 通过 chunked prefills 和 stall-free scheduling 改善 throughput-latency 折中；`DistServe` 和 `Splitwise` 将 prefill 与 decode 视为不同执行阶段，在资源映射和 goodput 层面进行更细粒度优化；`Preble` 进一步将 prompt sharing、prefix locality 与分布式负载均衡联合起来考虑；`Mooncake` 和同类 KVCache-centric 架构，则把 KV movement、disaggregation 与全局缓存池提升为一等系统对象。面向应用层，`Parrot` 通过 semantic variable 组织复杂 LLM application serving，也在更高层面处理“哪些 computation 可以复用”。

从系统形态看，`TwinPathServe` 与这一路线都包含 dual-pool、route key、family binding 和 admission decision 等结构，但两者所回答的问题并不相同。上述工作通常默认“只要一份 state 可以被当前 path 消费，主要问题就在于如何更好地调度、拆分、路由、搬运或放池”。本文则指出，在许多 `prompt_logprobs`、backend class 切换、carrier mismatch 或 typed API 场景中，问题发生得更早：系统尚未进入“如何调度这份命中”的阶段，就已经因为 consumer-side contract mismatch 放弃了原本可以成立的 reuse。

因此，`TwinPathServe` 虽然采用了 runtime control plane 和 dual-pool 结构，但其定位并非另一种 distributed 或 disaggregated scheduler。它首先是一套 contract-aware runtime design：通过 route key、family binding 与 `contract gate` 显式建模“为什么这次 hit 没有 materialize”，然后才讨论后续执行面选择。

## 7.3 shared-prefix execution：优化命中成立之后的执行

还有一类工作与本文在“共享前缀”这一表面词汇上最接近，但其真正关注的是命中成立之后如何更快执行。`Prompt Cache` 研究如何把可复用文本片段组织为模块化 attention reuse 单元；`Hydragen`、`FastTree` 等工作利用 shared prefix 或 tree-structured shared execution 优化 attention 和 decode；`Marconi` 则在 hybrid LLM 架构下重新讨论 prefix caching 的 admission 与复用边界。更近期的 prefix-aware kernel 路线，则继续沿着“共享已被识别后如何减少重复访存和执行”这一方向推进。

这些工作表明，shared-prefix execution 已经构成一条成熟且有竞争力的 systems 研究路线。但本文的切口并不在这里。`FalseNegativeKV` 并不重新设计一套 prefix-aware attention kernel，也不直接研究如何把 exact reuse 做得更快；本文关注的是，为何一些在参考执行条件下已经被证明可复用的 exact-prefix state，在当前 request 到来时甚至没有进入“可执行共享”的状态。换言之，shared-prefix execution 研究的是 `hit-established execution`，而本文研究的是 `pre-hit realization gap`。

这一差异决定了本文与 shared-prefix execution 路线的边界：前者关注命中成立后的执行优化，后者关注 `logical hit -> physical hit` 之间的运行时断裂。

## 7.4 非 exact-prefix 复用：PIC、multimodal caching 与 RAG cache fusion

位置无关缓存和更灵活的上下文复用，是近两年另一条重要的相关工作主线。`EPIC` 直接挑战 exact-prefix cache 的脆弱性，研究在 few-shot、RAG 等场景中如何做 position-independent context caching；后续多模态扩展如 `MPIC` 等工作，则把类似思想带入 multimodal serving；`CacheBlend` 则在 RAG setting 中研究 cached knowledge fusion，通过局部重算与 chunk 选择降低端到端推理成本。

从问题表面看，这些工作似乎与本文同样在研究“为什么系统没有复用已有内容”。但本质上，它们主动放宽的是 match 条件：当当前上下文不再是 exact prefix，甚至复用内容已经移动到中间位置或与检索结果交织在一起时，系统如何仍然构造可复用的 state。本文则将这一方向排除在外。`FalseNegativeKV` 的第一版严格坚持 exact 或 logical-exact reuse，不研究 arbitrary-position reuse，也不研究内容已非前缀时如何继续共享。

因此，本文中的 `selective replay` 不能被视为 PIC 的轻量变体。其作用不是让 non-prefix segment 也获得复用，而是在 prefix identity 已经成立的前提下，补齐当前 API 所需的 observation contract。相应地，`TwinPathServe` 也不是更灵活的 prefix cache，而是使本应成立的 exact-prefix reuse 不再在执行前被 consumer contract 中断的 runtime substrate。

## 7.5 state restoration、hybrid cache 与外部 KV 层：更换状态介质

还有一类近邻工作聚焦状态介质本身。`HCache` 研究如何用 intermediate activations 做更快的 state restoration；`Apt-Serve` 明确把 `KV + hidden cache` 组织为 hybrid cache，并研究由此带来的调度与恢复收益；`LMCache` 和同类工作则把 KV 暴露成跨引擎、跨层级的外部缓存层或服务化数据面，研究 offload、load、跨引擎共享与 memory hierarchy 的协同。更进一步的工作，例如把 KV cache 主动扩展为采样或推理轻量表示的方案，则尝试让缓存承担比标准 attention state 更广的语义角色。

这些工作与本文最容易混淆之处，在于 `prompt_logprobs` sidecar 容易被理解为另一种 hidden cache。本文所主张的并不是新的 state medium，也不是重新设计整个 cache hierarchy；`prompt_logprobs selective replay` 中的 sidecar 只承担一个严格受限的角色，即在 `ObservationCompleteness` 条件下，为当前 request 补齐 prompt-level observation 所需的最小伴生状态。它既不试图替代 KV，也不试图扩展成新的 activation cache 系统。

同样，`TwinPathServe` 的 dual-pool 结构也不依赖跨池共享 KV block 或外部 cache layer。第二类 bridge 的恢复方式不是“把一份 state 横向搬到更多不兼容 path 上”，而是“让本应在兼容执行类中消费的 family 回到兼容执行面”。因此，本文与 `HCache / Apt-Serve / LMCache` 等 medium-centric 路线保持清晰区分。

## 7.6 workflow continuity、augmented inference 与 agent serving

随着 tool-rich、multi-agent 和 RAG-enhanced workload 变得越来越常见，另一条快速增长的近邻工作开始关注 state 在复杂 workflow 中如何被保留和重新利用。`InferCept` 面向 augmented LLM services 研究生成被工具或外部调用截停后的高效继续执行；`KVFlow` 把 prefix cache、workflow structure 与 multi-agent request tree 放到一起，研究 eviction、prefetch 与 workflow-aware state management；其他 workflow continuity 路线也在持续探索如何维持跨步骤上下文连续性。`AugServe`、`RAGServe` 等系统，则更强调 augmented inference 场景中的调度、队列与质量-时延折中。

这类工作说明，state 的保存与继续利用本身在 agentic workload 中就是系统问题，但其矛盾焦点与本文不同。workflow continuity 研究的是 state lifetime 和 future reuse opportunity，即系统如何在多步交互、tool pause 或 retrieval loop 中不把有价值的 state 过早丢掉。`FalseNegativeKV` 研究的则是另一类问题：state 并未丢失，它甚至已经在 reference execution 下存在，但当前 request 仍然无法把它兑现为自己的 `physical hit`。这不是 retention failure，而是 usability failure。

因此，本文将问题边界限定在 `engine-local consumer compatibility`，而不扩展到跨步骤状态延续的更广泛议题。

## 7.7 feature-specific capability boundary 与工程讨论

除了正式论文，开源 serving runtime 的实现和文档也提供了重要的工程语境。`vLLM` 已在官方 guide 与源码中显式暴露出若干 capability boundary，例如 `prompt_logprobs` 请求为了保证 observation semantics 会跳过 prefix cache；在 `SGLang` 中，`input_embeds`、部分 multimodal content path、deterministic 或 backend-specific path 也会要求关闭 radix cache，或显式弱化 cache compatibility。这些现象说明，现代 serving runtime 中“state 已存在但当前 path 不允许直接消费”的边界并非孤立工程事故。

但本文与这类工程讨论的关系是承接而非重复。工程 issue、PR 和文档说明可以证明 mixed-API mismatch 已经被实际感知，却通常停留在 feature-local explanation：为什么某条 API 不能读 cache，为什么某个 backend path 需要保守关闭某项优化。本文试图补上的层级，是把这些分散边界上升为一个统一的问题类：`false-negative exact hit`，并进一步给出 measurement surface、`reuse contract`、`SafeReuseContract` 和 `bridge operator` 这套系统化语言。

开源系统 capability boundary 为本文提供了强有力的问题存在证据，但不能替代本文的学术贡献。本文真正填补的是这些工程现象之间缺失的中介层：统一命名、统一归因和统一恢复结构。

## 7.8 与本文的边界

综上，本文与现有相关工作的差异可以收束为四个层面。第一，本文并不讨论如何更快地利用已经可消费的 prefix hit，因此不同于通用 serving runtime、shared-prefix execution 与 distributed routing 这三条主线。第二，本文并不讨论如何让不再是 exact prefix 的内容继续复用，因此不同于 PIC、multimodal PIC 与 RAG cache fusion 路线。第三，本文并不重新设计 state medium，因此不同于 hidden cache、hybrid cache 和 external KV layer 路线。第四，本文并不研究如何在 agent / RAG workflow 中延长 state 生命周期，因此不同于 workflow continuity 与 augmented inference 路线。

本文所填补的空白更具体，也更克制：在 exact 或 logical-exact reuse 已经被 reference execution 证明成立的前提下，现代 serving runtime 仍缺少一套显式语言来表达 producer state、consumer requirement、mismatch reason 与 safe bridge 之间的关系。`FalseNegativeKV` 因此将 logical hit 在计算开始前丢失这一 runtime pathology 提升为一个独立的 systems 问题类。
