# 2 背景与动机

理解本文问题需要先区分前缀状态的存在与前缀收益的兑现。对在线推理系统而言，前缀复用是否成立，不仅取决于共享前缀是否已经建立身份对应关系，还取决于当前请求能否在其返回语义、执行路径和正确性约束下合法消费既有状态。本章据此界定研究边界，并说明为何异构接口负载会系统性暴露 `logical hit` 与 `physical hit` 的脱节，以及为何后续分析必须建立在请求级观测之上。

## 2.1 exact-prefix reuse 在 serving runtime 中如何产生收益

对于 decoder-only 大模型推理而言，长 prompt 的主要代价集中在 prefill 阶段。只要多个请求共享足够长的输入前缀，系统就有机会跳过这段已经完成的前向计算，从而降低首 token 延迟并提高单位硬件上的有效吞吐。因此，exact-prefix reuse 已经成为现代推理框架的基础能力，而不是附属优化。

在本文关心的 setting 中，前缀复用的收益并不是抽象地来自“系统里某处存在一份状态”，而是来自运行时真正把那份状态兑现为当前请求的可见命中。只有当两件事同时成立时，用户才会观察到复用收益：其一，某个参考执行已经建立了可复用的前缀状态；其二，当前请求在自己的执行路径上被允许直接消费这份状态。前者回答“状态是否存在”，后者回答“状态是否能被当前请求用上”。本文之后所说的 `physical hit`，正是第二层真正被兑现出来的那部分收益。

本文仅讨论 exact-prefix reuse，即当前请求与参考请求在可复用部分上共享同一语义前缀，且这部分前缀已经在某个执行条件下被证明可以稳定复用。本文不讨论 arbitrary-position reuse、distributed cache pool 或 cross-model sharing，因为这些问题会显著扩张命题边界，也会改变后续测量与系统设计的成立条件。

## 2.2 现有系统默认优化的是“可直接消费的物理命中”

现有推理框架对前缀复用的处理大致遵循同一思路：系统先根据输入前缀建立身份对应关系，再尝试让当前请求沿默认执行路径直接消费这份已计算状态；如果这一步成功，相关 prefill 计算就被跳过，并表现为可见的命中收益。这个流程在工作负载单一、请求语义稳定时非常有效，因为“前缀已经算过”与“当前请求仍然可以按原路径消费它”往往是同时成立的。

但这一处理方式隐含了一个很强的前提：一旦前缀身份相同，当前请求就仍然能够以与参考请求相同的方式消费那份状态。这个前提在现代 mixed-API serving 中并不总成立。当请求的返回语义、输入载体或执行后端发生变化时，系统未必还能把“已经存在的状态”自动转化为“当前请求可获得的收益”。换句话说，现有系统更擅长回答“这段前缀的 identity 是否已经建立”，却不总能回答“这份既有 state 是否仍然满足当前请求的消费条件”。

前缀缓存中的 hash 主要解决的是身份定位问题，即当前请求的前缀是否与某个已计算前缀对应到同一索引键；但真正发生复用，还要求当前请求能够在自己的返回语义、执行路径和正确性约束下合法地消费这份状态。因此，hash 命中是复用成立的必要条件，而不是充分条件。本文所研究的问题，正是出现于“参考执行已经证明某段前缀可复用”与“当前请求实际兑现了这份复用收益”之间发生脱节的时候。

## 2.3 什么情况下会出现 `false-negative exact hit`

为避免把所有 prefix miss 都纳入同一问题范畴，本文将 `false-negative exact hit` 限定为同时满足以下四个条件的情形。

1. 存在可参照的参考执行条件。更具体地说，必须存在同一 `prompt family` 下的参考请求，并且该参考请求已经证明共享前缀在某个执行条件中可以稳定复用。若某段前缀从未被计算过，或者从未在任何执行条件下被成功复用，那么它属于普通未命中，而不属于本文讨论的问题。
2. 当前请求与参考请求共享相同的可复用前缀。这里的“相同”并不要求所有请求属性一致，而是要求二者在可复用部分上语义等价。若当前请求的前缀本身已经变化、被截断，或者混入了不同 family 的内容，那么它也不属于本文的讨论范围。
3. 当前请求变化的是消费条件，而不是前缀本身。造成变化的可以是额外的 observation 需求、不同的输入载体，或者不同的 backend compatibility class；但这些变化不应改变“前缀已经被计算过”这一事实，而只是改变系统是否允许当前请求直接利用那份状态。
4. 当前请求没有兑现对应的复用收益。也就是说，在参考条件下可稳定获得的前缀命中，在当前请求中显著下降，甚至直接归零。只有在这一条件成立时，我们才说系统出现了“本应复用却未复用”的运行时失效。

这四个条件共同给出了本文的问题边界。如果只是哈希表中找不到条目、前缀此前从未计算过，或者请求本身的共享前缀已经不再成立，那么问题就应当被视为普通 miss，而不是本文要研究的 `false-negative exact hit`。明确这组条件的意义在于，将“状态不存在”与“状态存在但未被兑现”严格区分开来，从而避免把本文的问题退化为一般意义上的 cache key 或 lookup 失败。

## 2.4 为什么现代 mixed-API 负载会系统性触发这种脱节

今天的在线推理负载正在快速复杂化。一个看似共享相同前缀的请求，可能与参考请求具有不同的返回语义，例如需要额外返回输入 token 的概率信息；也可能采用不同的输入形态，例如嵌入向量或多模态载体；还可能被分配到不同的执行后端，以追求不同的吞吐、延迟或确定性目标。此时，“前缀是否相同”仍然重要，但已不足以单独决定系统能否兑现复用收益。

这意味着，现代推理系统面临的挑战已经不只是“如何提高缓存命中率”，还包括“如何防止逻辑上应当成立的复用在运行前悄悄坍塌”。如果系统只能看见最终有没有发生实际命中，就会把两类本质不同的情况混为一谈：一种是前缀状态根本不存在，另一种是前缀状态明明已经存在，却因为当前 consumer path 的消费条件变化而没有被利用。前者属于传统意义上的 miss，后者则更隐蔽，也更容易被现有运行时统计所忽略。

需要指出的是，这类脱节并不意味着现有系统的保守策略不合理。在许多路径上，运行时主动禁用或跳过 prefix reuse，恰恰是在维护正确性边界。更深层的问题在于，系统缺少一套显式语言去区分三件事：哪些状态已经存在，哪些状态当前不可直接消费，以及哪些不可直接消费的情形仍可被安全 bridge。正是在这一点上，现代 mixed-API serving 为本文问题提供了系统性土壤。

## 2.5 动机示例：共享前缀由稳定命中转为零收益

在 `vLLM` 的受控测量中，共享前缀长度约为 `1098` 个 token，而系统中稳定出现的命中收益为 `1088` 个 token。该差异并非测量误差，而是底层缓存粒度所致：系统按固定 block 大小统计可复用部分，因此只有完整对齐的部分才能被兑现为命中收益。这个细节已经说明，前缀复用并不是“前缀相同即全额生效”的简单映射，而受到运行时语义约束。

在这一基础上，我们进一步观察到两个更强的现象。第一，同一组共享长前缀的请求在普通文本生成场景下可以稳定获得 `1088` 个命中 token；但当请求仅额外增加一项 observation 需求，即返回输入 token 的概率信息时，这部分命中会直接降为 `0`。第二，同一组共享前缀在一种执行后端中可以稳定复用，但切换到另一种 backend compatibility class 后，前缀收益也会完全消失。两种现象表面上属于不同路径，实质上却共同指向同一个事实：前缀状态并没有消失，消失的是系统把这份状态兑现为当前请求收益的能力。

这一现象表明三点。第一，问题发生在计算开始之前，而不是解码阶段的局部性能波动。第二，问题的核心不在于系统无法定位前缀，因为参考请求已经证明相同 family 的可复用前缀确实存在。第三，仅凭聚合层命中率不足以解释此类现象，因为同样的“零命中”既可能来自状态不存在，也可能来自状态存在但当前请求不能直接消费。

## 2.6 请求级测量的必要性

如果没有一套能够在请求级别区分 `logical hit` 与 `physical hit` 的观测面，本文讨论的问题就只能以零散 feature case 的形式存在。对于 `prompt_logprobs`，我们只能看到“开启某个返回选项后命中掉了”；对于 backend mismatch，我们只能看到“换一个执行类后复用没了”；但系统无法进一步回答更关键的问题：这些现象是否属于同一个问题类，它们到底丢了多少本应兑现的收益，以及哪些 case 只是普通 miss，哪些才真正构成 `false-negative exact hit`。

因此，后续分析必须建立在请求级 trace、family-level reference/treatment 配对，以及能够显式区分“状态不存在”和“状态存在但未被兑现”的度量定义之上。只有在这一前提下，`false-negative exact hit` 才能从直觉现象转化为可测量、可复核、可归因的系统对象。下一章将在此基础上给出本文的观测面、度量定义和核心测量结果。
