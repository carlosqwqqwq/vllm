# 6 实验评估

第三章已经证明 `false-negative exact hit` 可以被严格测量，第五章则给出了 `TwinPathServe` 的 `soundness-first` 系统设计。在此基础上，本章进一步评估两类 bridge 是否已经完成原型化、是否能够恢复命中，以及当前证据能够支撑哪些结论。相较于早期版本，本文还将 `W1` 的公开 prevalence 证据从单一 `MT-Bench` 扩展到了 `LongBench` 长上下文负载，以缩小问题表征与证据边界之间的差距。

本章的评估围绕四个问题展开。第一，`prompt_logprobs selective replay` 是否已经在真实 runtime 中把 `W1` 暴露出的 observation mismatch 恢复为可消费的复用收益。第二，sidecar、boundary replay 与 replay budget 是否构成该机制的必要条件。第三，`TwinPathServe` 的最小双池原型是否已经能够把 `W5` 中的 backend mismatch family 从 throughput-oriented 路径重新送回兼容执行面，并形成稳定 family binding。第四，在这些机制级结果之外，当前尚有哪些大规模评测尚未闭环，从而限制了更强系统性结论的提出。

## 6.1 评估目标与证据边界

第三章表明，`W1` 与 `W5` 都已经给出强测量证据，足以证明 `false-negative exact hit` 不是零散 bug，而是可以在 request-level trace 上被严格识别的系统现象。本章则进一步评估：围绕这两类强 mismatch，P1 prototype 是否已经从系统设计推进到可运行机制。因此，本章的直接评估对象不是一般性 prevalence，而是第五章两类 bridge 的可执行性、恢复能力与边界条件。

因此，本章将 claim 边界固定在 prototype 级别。凡是已经有原型、trace 和对照结果支撑的结论，本章予以明确陈述；凡是仍需更大规模公开 workload、完整 baseline matrix 或 `p95/p99 TTFT` 支撑的结论，则保留为当前边界之外的内容。这样的处理方式有助于保持评估章节与现有证据的一致性。

## 6.2 原型、基线与指标

当前评估基于单机 `vLLM` 原型，模型为 `Llama-2-7B-Chat`。与第三章相同，所有结果都建立在 request-level trace 之上；但与第三章不同的是，本章除了 `lookup` 结果外，还进一步使用了 replay 与 route 相关 trace，以观察 bridge 是否真正触发、是否把 `physical hit` 转化为当前请求可消费的收益。对 `prompt_logprobs selective replay` 而言，本章重点观察 `physical_hit_tokens`、`effective_reuse_tokens`、`prompt_logprobs_use_sidecar`、`sidecar_key_ready` 与 phase wall time；对 `TwinPathServe` 而言，本章重点观察 `selected_pool`、`route_reason`、family binding 行为与恢复后的 `physical_hit_tokens`。

本章使用三类互补的评测设置。第一类是 `W1` 的公开 prevalence 测量，当前覆盖 `MT-Bench W1` 与 `LongBench W1`，用于再次确认 `prompt_logprobs` 的 false-negative 规模不是单一 benchmark 特例。第二类是 `prompt_logprobs selective replay` 的 dedicated validation：先由一次 `prompt_logprobs` 请求建立 sidecar，再由第二次同类请求作为 probe 消费 sidecar 并触发 boundary replay。第三类是 `TwinPathServe` 在公开 workload 上的双池 formal evaluation，当前固定在 `MT-Bench W5`，用于确认 backend mismatch 不只在 smoke setting 中成立，而是在公开请求上也能恢复 degraded hit。

除 `TTFT`、phase wall time 与 throughput 指标外，当前评估还显式引入 `realization_efficiency` 与 `bridge_cost` proxy，用来回答“多少 latent hit 最终被兑现成当前请求可消费的收益”以及“系统为此付出了多大 bridge 代价”。这组指标的作用，不是替代传统时延吞吐指标，而是把第五章的 `realization policy` 从设计语言落成可以被实验直接检验的量化对象。

这三类设置虽然尚未全部收束进同一个 public runner，但并非三套彼此独立的证据口径。三者共享同一套 request-level instrumentation、同一套 family pairing 规则、同一组 `logical hit / physical hit / false_negative_reason` 语义，以及同一条 `reuse oracle -> contract gate -> realization policy -> bridge / route -> fallback` 控制链。因此，当前评估虽未形成单一的大规模统一跑表，但已经不再是互不相关的局部结果拼接。

这组设置也决定了本章的基线含义。对第一类 bridge，`status quo` 基线是默认关闭 selective replay 的 `prompt_logprobs` 路径；对第二类 bridge，基线不是跨 backend 共享 state，而是让 family 保持在 throughput-oriented 执行面中运行。换言之，第二类 bridge 的恢复方式始终是 route back to a compatible pool，而不是 unsafe cross-backend state consumption。这一点不仅与第五章的 `SafeReuseContract` 一致，也直接关系到本文所强调的安全性边界。

为避免统计口径混淆，本章统一采用如下约定：像 `1088`、`148` 这样的数字默认表示单 request 或单 family 的平均命中；像 `8704`、`8576`、`592`、`256` 这样的数字则表示按配对 probe 聚合后的总量。前者更适合直观展示单个 family 的命中水平，后者则与第三章中的 `logical_hit_tokens` / `physical_hit_tokens` 记账方式严格对齐。

## 6.3 Bridge I：`prompt_logprobs selective replay` 已在 formal validation 中恢复 `W1` 的 false-negative hit

`prompt_logprobs selective replay` 已经不再只是设计构想，而是具有真实 runtime 闭环的第一类 bridge。这里需要区分两层证据：`MT-Bench W1` 与 `LongBench W1` 的公开 prevalence 结果共同说明，`prompt_logprobs` 在短上下文与长上下文公开 workload 上都会大规模抹去 `logical hit`；而下面的 dedicated validation 则进一步回答，当 sidecar 与 boundary replay 同时具备时，这部分 miss 是否能够被恢复。在 dedicated validation 中，baseline 与 prototype 的 probe 请求只在是否开启 selective replay 上不同，其余 prompt 语义保持一致。下表报告 `32` 个 probe request 的平均结果。默认路径下，probe 请求仍然完全无法兑现已有前缀复用；开启 bridge 后，同类 probe 请求已经被恢复为高 `effective reuse`。

| 配置 | probe 请求数 | 平均 probe `physical_hit_tokens` | 平均 probe `effective_reuse_tokens` | `prompt_logprobs_use_sidecar` | 平均 phase wall time |
| --- | --- | --- | --- | --- | --- |
| baseline off | `32` | `0` | `0` | `0 / 32` | `2.5772s` |
| selective replay on | `32` | `1088` | `1072` | `32 / 32` | `0.9765s` |

这组结果反映三点。第一，`1088 -> 1072` 并非实现误差，而是 selective replay 的设计语义：系统保留大部分命中前缀，但主动放弃命中边界的最后一个 cache block，并从该位置 replay 到 prompt 末尾。因此，`physical hit` 描述的是原本可见的命中上界，而 `effective reuse` 描述的是在 observation contract 下真正保留下来的复用量。按聚合 probe 记账，这对应于 `34816 -> 34304` 的恢复总量。第二，probe 请求的 `false_negative_reason` 从默认路径下的 `prompt_logprobs_skip` 变为 bridge 成功后的 `no_false_negative`，说明恢复结果已经从 false-negative miss 进入可消费路径。第三，phase wall time 从 `2.5772s` 降到 `0.9765s`，相对下降约 `62.11%`，速度提升约 `2.64x`，说明恢复收益不仅体现在 trace 字段上，也体现在端到端执行时间上。

对安全性而言，这组结果还有一层更深的含义。当前 bridge 不是简单地把 `skip_reading_prefix_cache` 从真改成假，而是在 sidecar 与 boundary replay 同时满足时，才把已有前缀状态恢复成当前请求可消费的 observation。与此对应的接口级冒烟结果也支持这一点：在同一 `LLM` 实例中顺序发送两次相同的 `prompt_logprobs` 请求时，方法版的第二次请求能够在成功返回 `prompt_logprobs` 的同时获得非零前缀命中，而关闭 bridge 的对照组第二次请求仍保持零命中。换言之，当前 prototype 至少已经证明：bridge 不会通过“丢字段”或“返回部分 observation”的方式伪造收益。更严格地说，这里已经闭环的是 dedicated validation 中的接口级 `observational completeness` 与显著时间收益；覆盖 top-k token、rank、logprob 数值逐项对齐的完整 output-equivalence matrix，仍有待后续实证工作补齐。

## 6.4 消融结果：sidecar 和 replay budget 都是必要条件

这一节的消融实验表明，`physical hit` 并不足以自动转化为收益。由于本节的目标在于分离 sidecar 与 replay budget 的机制作用，而不是报告最终性能上界，因此这里采用较小的 repeated `prompt_logprobs` 验证负载，并报告 `8` 个 probe request 的聚合结果。实验结果显示，系统必须同时满足 sidecar 可用、boundary replay 可执行以及 replay 成本在预算内，才能把 `physical hit` 转化为真正的 `effective reuse`。

| 配置 | 聚合 probe `physical_hit_tokens` | 聚合 probe `effective_reuse_tokens` | 相对 baseline 的 phase wall time 变化 |
| --- | --- | --- | --- |
| `baseline_off` | `0` | `0` | 基线 |
| `sidecar_only` | `0` | `0` | `+5.8%` |
| `replay_budget_25` | `8704` | `0` | `+0.3%` |
| `replay_budget_256` | `8704` | `8576` | `-73.1%` |

这张表说明了两个事实。第一，sidecar 单独存在并不会自动带来收益；如果没有真正的 replay，系统仍然无法把 observation mismatch 转化为可消费命中。第二，`physical hit` 也不等于 bridge 成功。在 `replay_budget_25` 这一组中，调度器已经观察到与 full budget 相同的物理命中，但由于真实所需 replay token 数超过预算，系统最终仍把 `effective_reuse_tokens` 压回 `0`，也没有出现实质时间收益。这说明 `contract gate` 的预算判断并非装饰性开关，而是 bridge `soundness-first` 行为的实证体现。

这组消融结果把第五章中的两个抽象判断转化为经验事实。其一，`ObservationCompleteness` 直接决定系统能否把命中恢复为当前 API 可消费的输出。其二，`VerifiedFallback` 不是附加补丁，而是系统在 sidecar 缺失或 replay 预算不足时主动拒绝不安全恢复的必要机制。由此，第一类 bridge 不再是简单的工程开关调整，而具有明确的条件结构。

从 `realization policy` 的角度看，这组消融还说明，并非每个 counterfactual hit 都值得被恢复。`replay_budget_25` 表明，哪怕逻辑上存在与 full budget 相同的 `physical hit` 上界，只要预算不足、`bridge_cost` 超出当前可接受区间，系统就应把该请求判为 `measure_only` 或直接 fallback，而不是把一次高成本 bridge 计为“成功恢复”。这也是本章同时报告 `realization_efficiency` 与 `bridge_cost` proxy 的原因。

## 6.5 Bridge II：`TwinPathServe` 已在 `MT-Bench W5` formal evaluation 中恢复 backend mismatch

第二类 bridge 的评估对象不是单个 worker 内部的 backend 切换，而是第五章定义的 twin-pool route。在当前 formal evaluation 中，系统包含 `reuse_optimized_pool` 与 `throughput_optimized_pool` 两个执行面；同一 family 先在 reuse pool 中建立 reference hit，然后分别比较 throughput baseline、第一次 `backend_mismatch` 路由与后续 `family_binding` 路由的结果。当前 `MT-Bench W5` 的公开结果如下。

| 路径 | 请求数 | `selected_pool` | `route_reason` | 平均 `physical_hit_tokens` | 平均 phase wall time |
| --- | --- | --- | --- | --- | --- |
| throughput baseline | `64` | `throughput_optimized_pool` | `forced_throughput_optimized_pool` | `93.00` | `0.6208s` |
| reuse reference | `64` | `reuse_optimized_pool` | `forced_reuse_optimized_pool` | `183.25` | `0.1474s` |
| twin-path backend route | `64` | `reuse_optimized_pool` | `backend_mismatch` | `183.25` | `0.0910s` |
| twin-path family binding | `64` | `reuse_optimized_pool` | `family_binding` | `183.25` | `0.0896s` |

这组结果表明，第二类 bridge 已经具备三层系统含义。第一，`TwinPathServe` 的 route policy 并非仅是 annotation；它确实把 backend mismatch family 从 throughput-oriented 路径重新送回了兼容执行面，并恢复到了 reuse reference 的命中水平。第二，恢复方式仍然遵守第五章的安全边界，因为系统并未尝试跨不兼容 backend class 直接消费同一份 state，而是把请求重新路由到能够合法消费该 state 的 pool。第三，family binding 不再是一次性 hard-coded 路由，而能够接管后续同 family 请求的稳定归属。

这里必须避免把控制面标签误写成 vanilla runtime 的原生真值。当前 `route_reason = backend_mismatch` 的含义，不是 throughput 池在本地 lookup 后自己报告了一个在线 skip reason；它表示控制面在 probe 到达之前，已经通过独立的 reference phase 建立了 family 级 compatible-hit evidence，并结合 backend compatibility metadata 做出了 admission 诊断。也就是说，route 标签依赖的是先验 reference 证据，而不是“先把请求路由过去，再用路由结果反过来证明路由是对的”。这也是为什么本章始终把 `W5` 的当前结论写成 compatible-reference recovery evidence，而不是 vanilla runtime 已经原生识别 backend mismatch。

从配对总量看，当前 `MT-Bench W5` 已经给出：

```text
logical_hit_tokens = 11728
baseline throughput physical_hit_tokens = 5952
baseline false-negative ratio = 0.492
twin-path backend physical_hit_tokens = 11728
twin-path family-binding physical_hit_tokens = 11728
twin-path backend recovery ratio = 1.000
twin-path family-binding recovery ratio = 1.000
```

从 `realization policy` 的语言看，这里对应的是 `realization_efficiency = 1.000`：控制面识别到的 latent exact hit 在 twin-path route 中被完整兑现成了当前运行条件下的 `physical hit`。这一定义之所以重要，是因为它把“命中恢复了”从口语描述收束成了严格的 realization 结果，也为后续把 backend route、observation bridge 与其他 bridge operator 放进同一评测口径提供了统一记账方式。

按 phase 平均值换算，`physical_hit_tokens` 从 `93.00` 提升到 `183.25`，相当于提升约 `97.04%`；backend probe 的 phase wall time 从 `0.6208s` 降到 `0.0910s`，相当于时延下降约 `85.34%`；family-binding probe 则进一步降到 `0.0896s`。这意味着第二类 bridge 已经不只是 smoke-level 方向性证据，而是开始具备可以进入论文主评测部分的公开 workload 结果。

## 6.6 两类 bridge 结果的含义与边界

把第 `6.3` 和第 `6.5` 节放在一起看，可以得到一幅比此前更完整的证据拼图。对第一类 bridge 而言，我们已经有 `MT-Bench` 与 `LongBench` 的公开 prevalence 结果证明 `prompt_logprobs` 会在两类公开 workload 上大规模制造 false-negative miss，也有 dedicated validation 证明 `prompt_logprobs selective replay` 的确能够把这些 miss 恢复成接近完整的 `effective reuse`，并带来显著时延收益；但它当前仍主要停留在 dedicated validation runner，而不是已经接进统一 public runner 的 full-method 结果。对第二类 bridge 而言，`TwinPathServe` 则已经跨过了这一步，正式在 `MT-Bench W5` 上展示了 contract-aware route 如何把 degraded hit 恢复到 reference-compatible path 的命中水平。

因此，两类 bridge 已经进入不同阶段的实证闭环。第一类 bridge 已经完成 runtime prototype 与 formal validation，第二类 bridge 已经进一步完成公开 workload formal evaluation。但需要同时指出，当前结果尚不能支持“backend mismatch 已在 vanilla runtime 中获得在线原生归因”这一更强表述；当前 route 仍然依赖 family 级历史证据、typed metadata 与 twin-path policy 的联合判断，这一点必须与第三章的 measurement 边界保持一致。

## 6.7 小结与局限

综合当前结果，本章已经能够给出三个结论。第一，`prompt_logprobs selective replay` 已经完成从设计到 runtime prototype、再到 `32-family` formal validation 的闭环：它不仅恢复了 `W1` 中的 false-negative miss，还通过消融证明了 sidecar、boundary replay 与 replay budget 共同构成桥接成功的必要条件。第二，`W1` 的公开 prevalence 证据已经从 `MT-Bench` 扩展到 `LongBench` 长上下文负载，说明 observation mismatch 的问题规模并不局限于单一公开 benchmark。第三，`TwinPathServe` 的第二类 bridge 也已超出 architecture 层面的描述；双池原型已经在 `MT-Bench W5` 的 `64` 个公开请求上把 backend mismatch family 重新送回兼容执行面，并形成可观测的 family binding。

本章仍然存在若干尚未闭环的部分。当前还没有覆盖 `W1/W2/W5` 多公开 benchmark 的统一主实验，也尚未把 vanilla、ad hoc per-feature fix 与 unified `TwinPathServe` 放进同一套完整 baseline matrix 中比较；`p95/p99 TTFT`、steady-state throughput、route fallback rate 与 overload behavior 也尚未在大规模 workload 上系统报告。对第一类 bridge 而言，当前 formal 结果仍来自 dedicated validation runner，而不是已经接进统一 public runner 的 full-method 路径；覆盖 top-k token、rank、logprob 数值对齐与长尾 fallback path 的 output-equivalence audit 也还没有完整展开。因此，当前评估已经足以证明“两类 bridge 均具有可执行性，且恢复方向正确”，但尚不足以支持更强的完整系统收益结论。
