# 目录结构与阅读方式

这份文档专门解释 `04_refined_ideas/` 这一层为什么要重新分目录，以及后面应该怎么继续维护。

## 1. 为什么要分子目录

随着调研继续深入，每个 idea 都会逐渐拥有不止一篇文档，例如：

- 方案总述
- 重合度与可行性评估
- OSDI/CCF-A 可投稿性评估
- 实验设计草案
- 代码改造路线图

如果这些文档都继续平铺在顶层，会出现两个问题：

- 顶层名字很多，读者很难先抓住“有哪些主线”
- 同一个 idea 的文档会被打散，后面的人很难顺着一条线深入

所以这次把结构统一成：

- **顶层看目录逻辑和横向决策**
- **子目录看单个 idea 的全部材料**

## 2. 当前目录逻辑

顶层文档负责三件事：

- `README.md`
  - 介绍这一层的定位、当前主线和推荐排序
- `directory_guide.md`
  - 说明为什么这样分目录，以及后续怎么维护
- `comparison.md`
  - 横向比较所有 idea，帮助决策

每个子目录负责一个 idea：

- `tierkv/`
- `affinitykv/`
- `qoskv/`
- `segmentkv/`
- `heterokv/`

每个 idea 子目录里的默认组织方式是：

- `README.md`
  - 该 idea 的主文档
- 其他补充文档
  - 可行性评估
  - 顶会适配评估
  - 风险分析
  - 未来若补实验或实现草案，也继续放这里

## 4. 建议阅读顺序

如果你是第一次看这一层，建议按下面顺序：

1. 看 `README.md`
2. 看 `comparison.md`
3. 进入最想继续推进的那个子目录
4. 先读该子目录的 `README.md`
5. 再读该目录中的评估文档

如果你是后续继续维护的人，建议按下面顺序：

1. 先判断新增内容属于哪个 idea
2. 若属于某个已有 idea，就放入对应子目录
3. 只有跨多个 idea 的文档，才放回顶层

## 5. 后续维护规则

后续补文档时，默认遵循下面规则：

- 不再把某个 idea 的新评估文档直接丢到顶层
- 某个 idea 的补充文档，优先放到对应子目录
- 顶层只放：
  - 总览
  - 结构说明
  - 横向对比
  - 真正跨多个 idea 的公共文档

## 6. 一句话总结

这层目录现在的设计原则是：

> 顶层负责“看全局和做决策”，子目录负责“沿单个 idea 深入”。
